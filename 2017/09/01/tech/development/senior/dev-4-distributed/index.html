CAP理论

  • CAP入门

  • CAP之P（分区容错性）是指什么
  • CAP为什么不能同时满足详解
  • CAP之可以不保证P（分区容错性）吗
  • CAP之CP和AP对比入门
  • CAP之CP和AP对比详解
  • CAP
    • C：强一致性：某个时刻读取所有节点的某个数据都是相同的（强一致性，不同于弱一致性和最终一致性）
    • A：高可用性：在有限的时间返回数据（不能出现等待延迟或者访问超时的情况）
    • P：分区容错性：网络故障出现分区后各个可用分区也能继续对外提供服务（而不是暂停服务去等待网络恢复）

ps：在分布式系统中，因为网络的复杂性，所以一定会出现因为网络故障导致分区的情况，如果此时不保证分区容错性，即系统暂停服务去等待网络恢复，这就违背了分布式系统中即使部分节点故障依然可以对外提供服务的设计初衷，所以分布式系统中一定要保证分区容错性

  • BASE理论入门
  • BASE理论详解
  • BASE和ACID理论对比
  • BASE
    • BA：基本可用
    • S：软状态（允许有短暂的中间状态）
    • E：最终一致性
  • ACID
    • A：原子性

    • D：持久性

ps：BASE理论的一致性是最终一致性，而ACID的一致性是强一致性

事务的分类

  • 本地事务（ACID）
    • 2PC
  • 分布式事务（CAP）
    • CP（刚性事务、强一致性，DTP模型）
      • 2PC和3PC（Seata-XA/AT）
      • JTA和JTS（Java-API）
    • AP（柔性事务，最终一致性、BASE模型）
      • 补偿型（同步）
        • TCC（Seata-TCC）
        • SAGA（Seata-SAGA）
      • 通知型（异步）
        • 确保通知型（本地消息表、MQ事务消息）
        • 尽力通知型（本地消息表、MQ事务消息）

ps：TCC和2PC的流程相似，TCC适用于应用接口，2PC适用于存储系统
ps：确保通知型需要保证接受者一定能收到通知消息，尽力通知型不保证但提供消息查询接口

事务的概念

  • TX：应用程序与事务管理器之间的接口协议
  • XA：事务管理器与资源管理器之间的接口协议
  • AP：应用程序
  • RM：资源管理器
  • TM：事务管理器
  • 2PC：两阶段提交（Two-Phrase-Commit）
  • 3PC：三阶段提交（Three-Phrase-Commit）
  • TCC：Try、Confirm、Commit
